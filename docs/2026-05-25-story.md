# The Collaborative Gherkin Story
## How an AI moved through a problem space — from blank canvas to production-ready tool

*This document traces the full development arc of Collaborative Gherkin, told through the sequence of planning sessions and code changes made between 16 May and 25 May 2026. It is written as narrative input for a NotebookLM podcast, so the reader can hear the AI reason its way from first principles to a finished product.*

---

## Part One: The Blank Canvas (16–17 May)

The project began with a simple scaffold: Next.js, Tiptap, and a skeleton database. The opening question was deceptively hard — what does a collaborative Gherkin editor actually mean at the data model level?

Gherkin is a structured language. "Feature", "Scenario", "Given", "When", "Then", "And", "But" — each keyword has rules about what can legally follow it. A plain text editor would let you type anything. That's not the point. The AI's first substantive decision was to treat each Gherkin keyword as a typed *block* rather than freeform text, and to encode the legal sequencing rules in a single module: `src/lib/gherkin.ts`. One function, `canFollow()`, would be the single source of truth for every structural question in the app. From that decision, everything else would derive.

Before a single feature was built, the AI wrote a complete formal specification — `SPEC.md` — covering the full BDD document model, block transitions, editor behaviour, and export format. That spec was then split into per-section files under `spec/`. The lesson here is visible in the commit history: spec first, code second, tests as verification of the spec.

The first working version arrived on 17 May: a session list, a collaborative editor where multiple users could type simultaneously, and a WebSocket server to synchronise the shared document via Y.js. It was raw, but the architectural backbone was already in place.

---

## Part Two: Teaching the Editor its Own Rules (17–19 May)

With the scaffold running, the AI turned to the Gherkin rules themselves. The early days were about edge cases in the `canFollow()` logic — should `Given` be allowed directly after `Then`? The initial answer was no. A planning session on 18 May examined the spec change required to allow it, traced every place in the codebase where that assumption was encoded (unit tests, E2E test comments, the rule table itself), and produced a surgical plan: change one line in `gherkin.ts`, update the test arrays, remove two stale comments. Nothing more.

This pattern — find the single canonical place, change only that place, verify the ripple — became the characteristic working style.

Next came the richer editing features. On 19 May, the AI tackled four things in a single day:

**Image nodes.** A plan was written for adding image blocks to the Gherkin document model. The challenge: images are structurally leaf nodes — they don't produce Gherkin text — so the export and keyboard handling needed special cases. The AI planned the node type, the toolbar button, the Enter key behaviour after an image (which had its own plan: what block type do you insert when there's no previous Gherkin keyword to infer from?), and the edge case where a `RangeError` would crash if Enter was pressed on an image.

**Markdown export.** The spec required a plain-text export that a user could paste into Jira. The AI planned the serialiser in `gherkin.ts`, the renaming of the button from "Export" to "Export TXT", and the quirk that markdown tables needed blank-line wrapping to render correctly.

**Gherkin indentation.** Gherkin conventions indent `Given`/`When`/`Then` under their `Scenario`, and `And`/`But` further still. This is purely visual — CSS, not structure — but the plan was careful to note that the indentation should be expressed via the existing `data-gherkin-type` attributes already on each block, requiring no schema changes.

**Data tables.** The most complex feature of the week. Gherkin data tables live inside step blocks as pipe-delimited rows. The AI planned a custom Tiptap node, a floating toolbar for table management (add row, add column, delete), and the alignment of the table's left edge with the step text above it. The toolbar went through two iterations in a single afternoon — first as buttons inside the table, then as a floating icon toolbar that appeared only when a cell was focused.

By the end of 19 May the editor had image support, markdown export, Gherkin indentation, and data tables. What it didn't have yet was a visual identity.

---

## Part Three: Making It Look Like Something (19–20 May)

Up to this point the app had a developer's dark theme — amber and violet on near-black. On 19 May a full visual overhaul replaced it with a dark editorial aesthetic. Then, almost immediately, a second overhaul replaced *that* with the NSW Government Design System.

The NSW Design System decision is interesting. The app was being built for a government team. The AI's plan on 20 May worked through every element of the existing UI, identified which CSS selectors were safe to change (the shell: home page, session header, forms) and which must not be touched (the Gherkin editor itself, with its `.gherkin-*` class names that E2E tests depended on). The plan said, explicitly: "Do not rename any `.gherkin-*` class." The E2E suite was the protection layer that made bold UI changes safe.

---

## Part Four: The AI Coaching Feature (20 May)

The LLM review feature was planned as a first-class collaboration tool, not a bolt-on. The planning session on the morning of 20 May made a table of every decision before touching code:

- Where to store the prompt? Database — shared across users, changeable without a redeploy.
- Where to store the selected model? Database — same key/value table.
- How to expose editor content to the review action? A `useImperativeHandle` ref on the editor component, exposing a `getContent()` method.
- Which models? A hardcoded list of free OpenRouter models — Llama, Gemma, Mistral, Qwen.
- How to display results? A Markdown-rendered modal.

The plan traced every new CSS selector and confirmed none conflicted with existing E2E tests. Then it was built.

A follow-up plan the same morning addressed a UX gap: when a user closed the review modal, the result was lost. The fix was simple in principle — split one state variable into two (the cached result and a visibility flag) — but the plan was precise about every consumer of that state: the Escape key handler, the backdrop click handler, the close button, the new "View last review" button that appeared after the first review ran.

The naming shift from "Review with AI" to "Get AI Coaching" happened on the same day, along with a spinner animation during the request. Small touches, but they signal a shift in how the feature was framed: this was a coach, not a linter.

---

## Part Five: Refactoring for Testability (21 May)

By 21 May the app was feature-rich but architecturally tangled. The AI ran a systematic decomposition across three planning sessions.

**Extract Coaching module.** The LLM review logic was spread across two route handlers with no shared seam. The plan introduced a `Coaching` class with explicit dependency injection: `appSetting`, `fetch`, and `apiKey` passed in as constructor arguments. This meant tests could inject fake implementations without mocking at the module boundary. The typed errors — `RateLimitError`, `CoachingConfigError`, `CoachingRequestError` — made the error handling explicit and machine-checkable.

**Decompose GherkinEditor shell.** The editor component had grown into a monolith. The plan split it into focused modules: the toolbar, the block picker, the collaboration layer, the export utilities. Each module had a clear boundary. The AI noted this was a decomposition within the existing file structure, not a move to a new directory — avoid moving things unless the case is clear.

**Deepen Session module.** The session management code had similar issues: database calls mixed with business logic mixed with HTTP handling. A dependency injection seam was introduced so tests could inject a mock database without touching the real one.

The pre-populated scaffold canvas plan also landed on 21 May. New sessions had been opening to a blank white box — confronting for first-time users. The AI's solution was to seed five empty scaffold blocks (Feature, Scenario, Given, When, Then) on first sync via Y.js, and never reseed on subsequent loads. The plan's most careful section was the E2E update list: eight separate test files needed updating because they all started with a "click the Feature toolbar button" preamble that was now redundant, and count assertions throughout needed bumping because the seed blocks changed the baseline.

## Part Six: Confronting the Infrastructure Gaps (22 May)

On 22 May the AI took stock. A test coverage review had mapped every acceptance criterion in the spec against existing tests. Seven gap areas were found. The plan for addressing them was disciplined: two gaps (CSS indentation — untestable without visual regression tools; database-level constraints — covered by Zod at the route layer) were explicitly excluded with written justifications. The remaining five were addressed:

- A home page E2E suite that didn't yet exist
- HTTP contract tests for all four API routes
- Image feature E2E tests
- Data table keyboard navigation tests
- Import edge case unit tests

The same day, a project improvement recommendations document was written — not a plan for a feature, but a retrospective diagnosis of the whole codebase. Nine issues were identified and ranked by effort and risk:

1. A single `dev:all` command (trivial — one line in package.json)
2. The Prisma/PostgreSQL provider mismatch (structural — hardcoded `provider = "sqlite"` would silently fail in production)
3. Missing per-session authorization (blocked — no auth yet)
4. No CI pipeline
5. Duplicate `SessionSummary` and `SessionRecord` types
6. The WebSocket server had no type safety
7. NextAuth was on beta
8. No pre-commit hooks
9. Prisma v6 (outdated)

By end of day, items 1, 2, 4, 5, 8, and 9 were done. The CI pipeline was the most consequential — GitHub Actions running lint, typecheck, unit tests, and E2E tests on every push and PR.

---

## Part Seven: Building the Foundation Under a Running App (23 May)

The auth work was the most architecturally significant thing the AI did all week. On 23 May, having built a full-featured app on a placeholder `userId`, the AI added real authentication.

The planning session opened with a clear-eyed audit: every route was unauthenticated, HomeClient hardcoded a user ID, the POST sessions route trusted userId from the request body. The plan addressed this systematically:

- Magic link auth via Resend (no passwords to manage)
- JWT sessions (to avoid a naming conflict between NextAuth's `Session` model and the app's own `Session` Gherkin model)
- A test-only bypass provider using a shared secret, so E2E tests could authenticate without sending real emails
- Middleware gating every route

The E2E auth fix took multiple iterations. The first attempt used a browser-based CSRF flow that failed on CI. The second minted a JWT directly, bypassing the browser entirely. The third fixed a `__dirname` ESM incompatibility in GitHub Actions. Each failure produced a focused fix rather than a broader rewrite — the AI stayed in the minimal-change zone even under pressure.

Pre-commit hooks also landed on 23 May: Husky + lint-staged running `eslint` and `tsc --noEmit` on every staged TypeScript file before the commit. The plan noted one non-obvious detail: `tsc --noEmit` needs to run against the whole project, not just staged files, so it must be wrapped in `bash -c` to prevent lint-staged from appending filenames as arguments.

The Prisma 7 upgrade completed the day. It was a mechanical but multi-file change: new `prisma.config.ts` at the project root, new import paths across `db.ts`, `session.ts`, `coaching.ts`, a driver adapter for SQLite, explicit `prisma generate` steps (v7 no longer auto-generates after migrate), and a Node version bump in CI from 20 to 22.

## Part Eight: Polishing and Shipping (24–25 May)

With the infrastructure solid, the final days were about finishing touches.

A Y.js seed race condition was fixed: when the WebSocket server seeded a new document, a timing gap between document creation and client connection could cause the seed to run twice. The fix added a connection status banner so users could see whether they were live-syncing.

A custom sign-in page was built to match the NSW Design System — the auth flow now felt like it belonged to the same product as the editor.

Scenario codes were added to the spec and the E2E tests were aligned with them, giving every test a traceable requirement reference.

A Dependabot configuration was added to keep npm dependencies current going forward, with weekly update PRs.

A full dependency upgrade plan was written on 25 May — a careful, step-by-step sequence through every package with a major-version bump available (Vitest 2→4, TypeScript 5→6, Zod 3→4, Tiptap 2→3, Next.js 15→16), with one documented exception: NextAuth, pinned on beta by historical accident, skipped because the rollback cost exceeded the benefit.

## Themes

Looking back across the ten days, several patterns repeat:

**Spec before code.** Every feature started as a written description of the desired behaviour, often with the exact spec section number attached. Code was the implementation of the spec, not the spec itself.

**One canonical place.** When a rule existed — the Gherkin sequencing rules, the available LLM models, the prompt defaults — it lived in exactly one place. Plans that changed rules traced every consumer of that place before touching anything.

**E2E tests as a safety net for UI changes.** The `.gherkin-*` selector discipline meant the Gherkin editor could be restyled, decomposed, and refactored without fear, because the E2E suite would catch any behavioural regression.

**Explicit dependency injection over mocking.** The architecture review and the Coaching/Session module extractions all moved toward injecting dependencies as constructor arguments. This made unit tests faster, more reliable, and easier to read.

**Infrastructure as a feature.** The CI pipeline, pre-commit hooks, Dependabot config, and Prisma upgrade weren't treated as boring chores. They got planning documents, deliberate sequencing, and explicit risk assessments. A tool that can't be maintained isn't finished.

**Small commits, honest messages.** The git history reads like a conversation: "Fix E2E global setup CSRF failure on CI", "Fix __dirname in global-setup.ts for ESM (GitHub Actions)", "Fix Y.js seed race condition and add connection status banner". Each commit describes exactly one thing. When something broke, the blame range was always narrow.

---

## Where It Ended Up

Ten days from scaffold to production-ready: a real-time collaborative Gherkin editor with magic-link auth, AI coaching, data tables, markdown export, a government design system, a full CI pipeline, pre-commit hooks, Dependabot, and a formal spec for every feature. The app runs on SQLite in development and is documented and ready for PostgreSQL in production.

The AI didn't add features speculatively. It didn't refactor things that worked. It didn't mock the database when a real one was available. It wrote plans before writing code, updated tests before shipping features, and left the codebase cleaner at the end of each day than at the start.

That's the story.
