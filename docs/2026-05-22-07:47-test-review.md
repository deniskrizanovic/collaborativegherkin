# Test coverage review ‚Äî 2026-05-22

Analysis of whether every acceptance criterion in `spec/` maps to a test somewhere (unit or e2e).

### What counts as adequate coverage?

Unit tests are adequate when the spec criterion *is* the behaviour of a pure function ‚Äî e.g. `canFollow()` returning true/false, or `exportToText()` producing a specific string. Testing the function directly exercises the criterion exactly.

Unit tests are **not** adequate when the spec criterion is written in terms of an HTTP interaction (status code, response body, Zod validation at the route layer) or a UI interaction (button state, redirect, clipboard). In those cases a unit test on the underlying module confirms the business logic works in isolation but leaves the route handler, middleware, and integration untested. Those are marked as gaps even when a unit test exists.
