Problem. GherkinEditor.tsx is a 952-line shallow shell — its interface (one React component) gives callers almost no leverage over the internals, yet its implementation is nearly impossible to test because every concern is a closure.

Solution. Extract the keyboard handler into useGherkinKeyboard, the data table node view into its own file, the block picker into a standalone module, and the Y.js provider lifecycle into useCollabProvider; the editor shell composes them.

  • → interface shrinks: shell drops from ~950 lines to ~150; each extracted module has a small, stable interface

Problem. Each route handler directly constructs Prisma queries — DB coupling leaks across all three callers, and the session creation logic (Zod schema + default scaffold content) cannot be reused without going through HTTP.

Problem. SessionView.tsx mixes LLM settings fetching, two modal open/close state machines, and two duplicated Escape key listeners — its interface gives callers no leverage over these concerns individually.

Solution. Extract useLlmPanel hook to own all LLM panel state, settings fetch, and a single unified Escape handler; SessionView becomes JSX only.

  • → leverage: useLlmPanel testable with renderHook and a fetch mock, no DOM needed

Top recommendation

Deepen the LLM review module

It has the highest ratio of hidden complexity to interface — rate-limit parsing, DB-backed prompt loading, and OpenRouter coupling all live inside one route handler with no seam. A single LlmReview module behind a clean seam would collapse three scattered concerns into one testable place, and the two natural adapters (HTTP in prod, in-memory in tests) justify the seam immediately. Tackle this before the editor decomposition — it's smaller, the seam is well-defined, and the win in locality is immediate.
