Test Coverage Review

Analysis of whether every acceptance criterion in spec/ maps to a test. Generated 2026-05-22.

What counts as adequate coverage?

Unit tests are adequate when the spec criterion is the behaviour of a pure function — e.g. canFollow() returning true/false, or exportToText() producing a specific string. Testing the function directly exercises the criterion exactly.

Unit tests are not adequate when the spec criterion is written in terms of an HTTP interaction (status code, response body, Zod validation at the route layer) or a UI interaction (button state, redirect, clipboard). A unit test on the underlying module confirms business logic in isolation but leaves the route handler, middleware, and integration untested. Those are marked as gaps even when a unit test exists.

Spec 02

Gherkin Document Model

All canFollow() rules are fully covered in src/lib/gherkin.test.ts. The spec criteria are the return values of a pure function, so unit tests are adequate here.

Image blocks and data table blocks being exempt from canFollow() are not explicitly unit-tested, but this is a non-issue — canFollow() only operates on GherkinBlockType and the test suite exhaustively exercises every type transition.

Spec 04

API Behaviour

No HTTP-level API contract tests at all. The session routes (GET/POST /api/sessions, GET/DELETE /api/sessions/[id]) have zero coverage — only the Session module unit tests exist.

Missing for every route: correct 2xx status codes and response shapes; 4xx validation rejection (Zod errors surfaced correctly); 5xx error responses with the specified message strings.
