What counts as adequate coverage?

Unit tests are adequate when the spec criterion is the behaviour of a pure function — e.g. canFollow() returning true/false, or exportToText() producing a specific string. Testing the function directly exercises the criterion exactly.

Route handler tests (using NextRequest against the actual handler) are adequate for HTTP-contract criteria — status codes, response shapes, and Zod validation rejection at the route layer. These are distinguished from service-layer unit tests which leave the handler untested.

E2e tests are required for criteria written in terms of a UI interaction (button state, modal, redirect, clipboard). A route handler test alone does not satisfy a UI criterion.

Spec 05

Data Model Constraints

No tests at any level. CUID id generation, title length enforcement (≤ 200 chars, non-empty), createdAt/updatedAt auto-set, userId foreign key, and unique email constraint are all untested.

These are database-schema constraints enforced by Prisma/SQLite. Tests would require either a real (test) DB connection or a schema-validation harness, neither of which exists yet.
